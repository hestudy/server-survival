# Spec: 移动端 Web 版 v1 —— 模块化 + 触摸支持 + 响应式 HUD(生存模式)

> 本规格由规划会话产出。术语以 `CONTEXT.md` 词汇表为准;约束以 ADR-0001(与上游彻底分家)、ADR-0002(Vite + 平台适配层)为准。
> 触发标签:`ready-for-agent`(待仓库启用 Issues 后同步至工单)。

## Problem Statement

Server Survival 目前只能在桌面浏览器游玩:平移靠鼠标右键拖拽、缩放靠滚轮、查看详情靠 hover、相机靠 WASD,代码中没有任何触摸事件处理;UI 是四个固定宽度的角落面板加 13 个按钮的底部工具栏,没有一条媒体查询,手机屏幕上面板互相重叠、遮住画布。手机用户打开游戏后完全无法游玩。

同时,项目的长期目标是移植为微信小游戏,而现有代码是 20 个 `<script>` 标签、全局变量通信的单体结构(主文件近 4000 行,含 189 处 DOM 操作),既无法被小游戏环境复用,也无法为触屏改造提供安全的施工基础——历史回归(路由死循环、请求泄漏、重复计分)全部集中在与 DOM 纠缠的仿真逻辑里,且没有任何自动化测试兜底。

## Solution

分四个里程碑交付一个手机浏览器上完整可玩的生存模式:

- **M0 触控扭曲(丢弃式)**:在现有代码上硬接最粗糙的单指平移/双指捏合/tap,真机验证「连线玩法在触屏上是否成立」,结论记录后代码丢弃。
- **M1 模块化**:按「仿真核心 / 渲染层 / UI / 输入 / 平台适配层」拆分,ES modules + Vite 构建,Three.js/Tailwind 转 npm 依赖,GitHub Pages 改走 Actions 构建部署;拆分同步为仿真核心补上特征化测试。
- **M2 正式触控**:手势体系落地——tap=点选,双指捏合=缩放,单指拖拽一律=平移相机,移动服务必须先长按「抬起」(约 350ms,带反馈);连线沿用工具模态下先后点选两个服务。
- **M3 响应式 HUD**:小屏采用「顶部状态条 + 底部抽屉」布局,财务/健康/服务详情收进抽屉标签页;桌面宽屏断点下执行「桌面冻结」——布局与交互与现状保持一致。

## User Stories

1. As a 手机玩家, I want 用手机浏览器打开游戏就能进入生存模式, so that 我不需要一台电脑也能玩。
2. As a 手机玩家, I want 单指拖拽画布来平移相机, so that 我能用最自然的手势查看战场各处。
3. As a 手机玩家, I want 双指捏合缩放视角, so that 我能在全局观察和局部操作之间快速切换。
4. As a 手机玩家, I want tap 一个服务就看到它的实时详情(容量、健康度、修理费), so that 桌面端靠 hover 才能看到的信息我也能获取。
5. As a 手机玩家, I want 从工具栏选中服务类型后 tap 网格空位完成放置, so that 我能单手搭建架构。
6. As a 手机玩家, I want 在连线工具下先后 tap 两个服务来建立连线, so that 我能完成游戏最核心的架构搭建操作。
7. As a 手机玩家, I want 移动服务必须先长按将它「抬起」(有震动/高亮反馈)再拖动, so that 战斗激烈时我不会误把架构拖乱。
8. As a 手机玩家, I want 单指拖拽在没有抬起任何服务时永远只平移相机, so that 手势行为可预测、不需要猜。
9. As a 手机玩家, I want 顶部有一条常驻的紧凑状态条(资金/声誉/时间/RPS/当前事件), so that 我扫一眼就知道局势,不被大面板遮挡战场。
10. As a 手机玩家, I want 财务、服务健康、节点详情收纳在底部抽屉的标签页里, so that 我需要时拉出来看、不需要时画布最大化。
11. As a 手机玩家, I want DDoS 波次和随机事件的警告在小屏上依然醒目, so that 我能及时反应而不是莫名其妙输掉。
12. As a 手机玩家, I want 工具栏在窄屏上可横向滚动且按钮满足触摸目标尺寸, so that 13 种服务我都能准确点到。
13. As a 手机玩家, I want 在中端手机上获得稳定 30fps, so that 高 RPS 时操作不卡顿。
14. As a 手机玩家, I want 游戏在性能不足时自动降低画质而不是掉帧, so that 老一点的手机也能顺畅玩。
15. As a 手机玩家, I want 暂停/加速等时间控制在触屏上可用, so that 我能在压力大时暂停思考。
16. As a 手机玩家, I want 修理服务和开关自动维修在触屏上可操作, so that 服务掉血时我有完整的应对手段。
17. As a 手机玩家, I want 我的存档在版本升级后依然有效, so that 更新不会吞掉我的进度。
18. As a 手机玩家, I want 在小屏上看到沙盒/战役/教程入口被隐藏并提示用桌面端体验, so that 我不会误入没有适配的模式而留下坏印象。
19. As a 手机玩家, I want 以我的系统语言(9 种已支持语言之一)显示界面, so that 移动版不丢失国际化能力。
20. As a 桌面玩家, I want 宽屏下的布局、快捷键和鼠标交互与改版前完全一致, so that 我的肌肉记忆和游戏体验不受这次改造影响。
21. As a 桌面玩家, I want 改造期间的每个版本都保持可玩且无回归, so that 我不会成为重构的受害者。
22. As a 开发者, I want 仿真核心成为不依赖 DOM/渲染器/平台 API 的纯逻辑模块(随机性经注入的 RNG 提供), so that 路由和经济结算可以在无头环境下确定性地测试。
23. As a 开发者, I want 路由、经济结算、事件系统、存档序列化在拆分前先被特征化测试锁定, so that 重构改变行为时测试立刻报警。
24. As a 开发者, I want 存储、音频、输入事件源只出现在平台适配层, so that 小游戏移植时只需替换适配层实现。
25. As a 开发者, I want Vite 构建 + GitHub Actions 自动部署 Pages, so that 提交合并后线上版本自动更新。
26. As a 开发者, I want 先用 1–2 天的丢弃式触控扭曲在真机验证玩法手感, so that 最大的不确定性在大额投入前被消除。
27. As a 小游戏移植者, I want 仿真核心与渲染层之间有清晰的状态消费边界, so that 换渲染后端(three-platformize)时不需要动游戏逻辑。

## Implementation Decisions

- **项目定位**:移动端 Web 版是微信小游戏的跳板;所有分层决策以「适配层可整体替换」为准绳(ADR-0002)。
- **上游关系**:动工前最后同步一次上游 main 作为基线,此后永久分叉,上游改动仅做选择性人工搬运(ADR-0001)。
- **模块分层**:仿真核心(纯逻辑,注入 RNG)/ 渲染层(Three.js,单向消费仿真状态)/ UI 层 / 输入层(手势识别:原始 pointer 事件 → 语义意图)/ 平台适配层(存储、音频、输入事件源)。DOM 与 localStorage 禁止出现在仿真核心。
- **构建**:ES modules + Vite;Three.js 与 Tailwind 从 CDN 改为 npm 依赖;「克隆即玩」不再是承诺,`npm i && npm run dev` 取代之;Pages 部署改为 Actions workflow。
- **v1 玩法范围**:仅生存模式;沙盒/战役/教程在小屏隐藏入口并提示用桌面端。
- **手势体系**:tap=点选/查看;双指捏合=缩放;单指拖拽一律=平移相机;移动服务须先长按约 350ms「抬起」(触觉/视觉反馈)后拖动;连线沿用工具模态下的两次点选;hover 类信息(实时 tooltip、升级指示)全部并入 tap 选中后的详情展示。
- **小屏 HUD**:「顶部状态条 + 底部抽屉」模式;抽屉承载财务/健康/详情标签页;桌面与移动共用一套 DOM,靠断点切换,不做两套页面。
- **桌面冻结**:宽屏断点(约 1024px)下布局与交互保持现状,作为每个重构变更的回归基准;桌面 UI 重设计明确不做。
- **性能基线**:中端机稳定 30fps、旗舰 60fps;手段为 devicePixelRatio 封顶 2、请求粒子上限、自动降阶(关阴影/简化材质);兼容目标 iOS Safari 15+ 与近两年 Android Chrome。
- **存档兼容**:localStorage 键与序列化格式保持向后兼容。
- **国际化**:i18n 机制不动,仅为新 UI 增加键;`data-i18n` 扫描机制在共用 DOM 方案下继续适用。
- **施工顺序**:M0 触控扭曲(丢弃式,独立分支,产出为验证结论文档)→ M1 模块化 → M2 正式触控 → M3 响应式 HUD;每个里程碑桌面端保持可玩。

## Testing Decisions

- **原则**:只测外部行为,不测实现细节;测试从接缝打入,断言可观测结果(声誉/资金变化、请求终态、序列化往返一致),不窥探内部结构。
- **接缝 1(主)——仿真核心公共接口**:无头构造世界(放置服务、建连线、注入 RNG 与流量序列),`step(dt)` 推进,查询状态与序列化。路由、经济结算、事件系统、存档四大块的全部测试走这条接缝。拆分顺序为「先写特征化测试锁定现有行为,再重构」。
- **接缝 2(辅)——手势识别层**:输入合成 pointer 事件序列,断言输出的语义意图(点选/抬起/拖动/平移/捏合),覆盖长按阈值、tap 判定滑动容差、捏合与平移的冲突消解。
- **不设自动化测试**:渲染层、DOM UI、响应式布局——靠「桌面冻结」人工回归基准与真机验证。
- **工具**:Vitest;无既有测试可作先例,历史回归案例(路由死循环、请求泄漏、重复计分、存档丢财务状态)转写为首批回归用例。

## Out of Scope

- 微信小游戏本体移植(three-platformize 接入、小游戏 canvas UI、审核资质)——本规格只负责把跳板搭好。
- 沙盒、战役、教程三块体验的移动端适配(后续版本逐个补)。
- 桌面端 UI 的任何重设计或布局变更。
- 千元机/老机型的激进降级(2D 渲染等)。
- 向上游 pshenok/server-survival 回馈代码。
- 新玩法、数值平衡调整——本规格是纯工程改造,游戏行为以特征化测试锁定为准。

## Further Notes

- M0 是风险对冲步骤:若真机试玩证明连线玩法在触屏上不成立,后续里程碑应暂停并重新评估交互方案,而不是继续施工。
- 术语约定见 `CONTEXT.md`(点选/抬起/顶部状态条/底部抽屉/桌面冻结/仿真核心/平台适配层);实现与本规格冲突时,先修订文档再动代码。
- README 中指向上游 Pages 的「PLAY NOW」链接与零构建表述需随 M1 一并更新(ADR-0001/0002 的遗留事项)。
